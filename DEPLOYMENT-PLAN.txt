================================================================================
  EDGE CASES TO HANDLE
================================================================================

  1.  Stripe webhook arrives before frontend redirect
      → Frontend already trusts client-side "succeeded" status
      → Webhook is the source of truth for fulfillment

  2.  Stripe webhook replay / duplicate events
      → completePaymentByIntentId() checks if already COMPLETED, skips if so

  3.  User sees 404 for image thumbnails (variants not ready yet)
      → Add fallback: if _thumb.webp returns 404, show original image

  4.  User deletes listing with active promotion
      → Should cascade: cancel/expire promotion when listing deleted

  5.  JWT expires mid-session
      → Frontend detects 401 → redirects to login page

  6.  DynamoDB throttling under load
      → AWS SDK has built-in retry with exponential backoff
      → PAY_PER_REQUEST scales automatically

  7.  Large file upload fails
      → 5MB limit enforced server-side → clear error message to user

  8.  Concurrent promotion purchase for same listing
      → DynamoDB conditional write on promotionId prevents duplicates

  9.  S3 upload fails mid-transfer
      → Frontend should retry upload
      → Server returns error, doesn't mark upload as complete

  10. Stripe webhook signature verification fails
      → Return 400, log for debugging
      → Don't expose verification details in response

  11. User registers with same email twice
      → DynamoDB conditional write prevents duplicate
      → Returns clear "Email already registered" error

  12. Admin accidentally deletes wrong user/listing
      → DynamoDB PITR allows point-in-time restore
      → Consider soft-delete (status: DELETED) instead of hard delete

  13. CloudFront serves stale frontend after deploy
      → CI/CD pipeline must invalidate CloudFront cache after S3 sync
      → HTML files: Cache-Control: no-cache
      → JS/CSS: hashed filenames, long cache
